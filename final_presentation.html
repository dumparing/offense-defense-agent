Problem Statement

Security analysts manually repeat the same workflow for every binary they assess:

Identify target → Fuzz for crashes → Disassemble → Classify vulnerability → Report

Can an LLM autonomously drive this pipeline?

  • Plan which tool to use based on what it already knows
  • Execute security tools (GDB, objdump, nmap) via subprocess
  • Summarize raw tool output into structured findings
  • Remember across steps to avoid redundant work
  • Stop when analysis is complete
Goal: build the analysis phase of a red team agent — find and classify vulnerabilities, but deliberately stop before exploitation.

Agent Loop: Plan → Execute → Summarize → Remember

agent/agent.py — run_chain()

def run_chain(self, task, context, max_steps=5):
    """Autonomous multi-step analysis chain."""
    for step_num in range(1, max_steps + 1):
        # 1. Build context-aware prompt for the planner
        chain_task = self._build_chain_prompt(task, step_num, max_steps)

        # 2. Run one full agent turn (plan + execute + summarize + store)
        step_result = self.run(chain_task, context=persistent_context)

        # 3. Planner signals "done" → stop the chain
        if step_result.get("skill_used") is None:
            break

    # Build final report from accumulated memory
    return self._build_chain_report(task, steps)

Each turn:

  • Planner reads memory snapshot + skill catalog
  • Selects best skill + extracts arguments
  • Skill executes via subprocess
  • Summarizer compresses output
  • Memory updated with findings

Chain awareness:

  • Planner knows which skills were already used
  • Guided ordering: debug → disassemble
  • Stops when all relevant skills are complete
  • Hard limit prevents infinite loops

LLM Mode (Ollama running)

  • Llama 3 (8B) running locally via Ollama
  • No data leaves the machine
  • Understands context and natural language tasks
  • JSON output with retry on parse failure

Fallback Mode (no Ollama)

  • Keyword matching against skill catalog
  • Deterministic skill ordering for chains
  • Regex-based argument extraction
  • Pipeline never breaks — graceful degradation

Why memory matters: Without it, each step starts from zero. The planner needs to know "GDB already found 5 crashes" so it picks disassembly next, not another scan. Memory also prevents context bloat — raw GDB output is ~2KB, the summary is ~100 bytes.

targets/vuln_uaf.c — Use-After-Free

User *user = malloc(sizeof(User));
user->handler = normal_handler;
free(user);         /* freed */
char *new = malloc(sizeof(User));
fgets(new, ...);
user->handler(...); /* dangling! */

targets/vuln_heap.c — Heap Overflow

Account *acct = malloc(sizeof(Account));

Compiled with: gcc -fno-stack-protector -g — protections intentionally disabled for analysis

Architectural Safeguards

  • Skill registry boundary — can only run registered skills
  • Local-only LLM — no data leaves the machine
  • Subprocess timeouts — prevent resource exhaustion

Runtime Guardrails

  • Block system paths (/usr/bin/)
  • Warn on non-private IPs
  • Audit log of all decisions
  • Binary size limits (100MB max)

Safety Guardrails: Code

safety/risk_analysis.py — SafetyGuardrails

class SafetyGuardrails:
    BLOCKED_PATHS = ["/usr/bin/", "/usr/sbin/", "/bin/", "/sbin/"]
    MAX_BINARY_SIZE = 100 * 1024 * 1024  # 100 MB

    def check_binary_path(self, path):
        for blocked in self.BLOCKED_PATHS:
            if abs_path.startswith(blocked):
                self._audit("BLOCKED", f"System path: {abs_path}")
                return False, "Blocked: system binary"
        return True, "OK"

    def check_target_ip(self, ip):
        safe_prefixes = ["127.", "10.", "192.168.", ...]
        if any(ip.startswith(p) for p in safe_prefixes):
            return True, "OK"
        return True, "WARNING: not a private IP"

Failure Modes (Handled)

LLM Unavailable
Falls back to keyword planner + rule-based summarizer. Pipeline never breaks.
LLM Hallucinates Skill
Registry lookup fails → "skill not registered" → wasted step, not crash.
Adversarial Binary
Anti-debug (ptrace) can evade analysis. Out of scope for this prototype.

Key Design Decisions

Why local LLM?

  • Security tool output should never leave the machine
  • No API keys, no billing, no cloud dependency
  • Full control over model and context
  • Fallback ensures the system works without it

Why analysis only (no exploitation)?

  • Skill registry is a hard boundary
  • Adding exploit skills requires code changes + review
  • Lets us study autonomous offensive tooling safely
  • Analysis is the harder, more interesting problem

Why zero dependencies?

  • Runs anywhere Python is installed
  • No supply chain risk
  • stdlib has everything: subprocess, json, urllib, dataclasses

Why structured memory over raw context?

  • Raw GDB output: ~2KB per run
  • Structured summary: ~100 bytes
  • 4 turns of raw output would exceed Llama 3's context
  • Planner only needs "5 crashes found" not the full trace

Limitations & Future Work

Current Limitations

  • Simple fuzz inputs — not a real fuzzer (no coverage guidance, no mutation)
  • Anti-debug evasion — binaries with ptrace checks can evade analysis
  • Small model — Llama 3 8B sometimes misparses JSON or picks wrong skills
  • No exploitation — by design, but limits end-to-end red team utility
  • Static vuln patterns — relies on known dangerous function names

Future Work

  • Coverage-guided fuzzing — integrate AFL/libFuzzer for smarter crash discovery
  • Larger models — Llama 3 70B or fine-tuned security models
  • Exploit generation — controlled payload generation in sandboxed environments
  • Network attack chains — scan → identify service → check known CVEs
  • Multi-agent collaboration — separate red team and blue team agents
  • Symbolic execution — integrate angr for path constraint solving
Ethical note: Autonomous offensive tools require careful governance. Our safety analysis documents 10 risks and 6 architectural safeguards — but responsible deployment requires scope controls, human oversight, and institutional review beyond what a prototype can provide.

Conclusion

We built an autonomous LLM-driven agent that performs the analysis phase of offensive security — planning, debugging, disassembling, and classifying vulnerabilities across 4 vulnerability classes with 100% accuracy.
