Project Overview

Offensive security tasks like penetration testing, binary analysis, and exploit development require iterative reasoning, tool orchestration, and expert intuition. This project explores whether LLMs can act as autonomous agents that plan and execute such workflows.

We are building an LLM-driven offensive security agent that interacts with common security tools (scanners, debuggers, disassemblers) to complete structured attack-oriented tasks in sandboxed environments.

The agent follows a perception-action loop: issuing tool commands, interpreting results, updating its plan, and adapting strategy. Example tasks include debugging vulnerable binaries, identifying memory corruption primitives, and automated reconnaissance.

The goal is to evaluate whether LLM agents can reason over tool outputs and execute multi-step attack workflows in controlled environments, while understanding the risks of deploying them in offensive contexts.

The Perception-Action Loop

The agent loop runs one cycle per turn: perceive state, decide via LLM, act, observe result, update memory. Every step is logged to a decision trace for evaluation.

agent/agent.py

def run(self, task, context=None):
    # PERCEIVE
    snapshot = self.memory.get_context_snapshot()

    # DECIDE
    plan = self._plan(task, context, snapshot)

    # ACT
    result = skill.execute(**args)

    # OBSERVE
    summarized = self.summarizer.summarize_generic(
        skill_name, result)

    # UPDATE
    self.memory.record_action(...)
    self.memory.record_scan_results(...)

    return self._result(
        success=..., summary=...,
        memory_snapshot=..., trace=...)

The decision trace records every step: task received, memory read, planner decision, skill execution, summarization, memory update. This lets us reconstruct exactly why the agent chose a particular tool and how it interpreted results.

The agent takes three injectable components:

SecurityAgent(
    registry,      # available skills
    memory=memory, # shared state
    llm=llm,       # local Llama client
)

LLM Planner: Tool Selection

Each turn, the planner receives the task, skill catalog, and memory snapshot. It returns strict JSON specifying which tool to run, arguments, and reasoning.

Planner prompt (simplified)

TASK: "Scan the target for open services"
CONTEXT: {"target_ip": "127.0.0.1"}
MEMORY: {}
AVAILABLE SKILLS: [{"name": "network_scan", ...}]

Planner response

{
  "skill": "network_scan",
  "arguments": {"target_ip": "127.0.0.1"},
  "reasoning_summary": "Task requests
    reconnaissance; network_scan fits."
}

The selected skill is validated against the registry before execution. Invalid JSON or unknown skills produce readable errors.

Why local? Scan results and attack plans should not be sent to cloud APIs. Llama 3 via Ollama keeps all data on-machine. No API keys or billing.

Fallback: if Ollama is down, the agent falls back to keyword-based skill matching. The trace shows which path ran.

As we add more skills for the final, the planner will make increasingly complex decisions about tool sequencing.

Local LLM Client

Pure standard library HTTP client for Ollama at localhost:11434. Model configurable via OLLAMA_MODEL env var.

core/llm_client.py

def generate_json(self, prompt, system_prompt=None):
    raw = self._call_generate(prompt, system_prompt)
    parsed = self._try_parse_json(raw)
    if parsed is not None:
        return parsed

    # Retry with correction prompt
    correction = "Respond with ONLY valid JSON..."
    raw = self._call_generate(correction, ...)
    parsed = self._try_parse_json(raw)
    if parsed: return parsed

    raise OllamaJSONError("Failed to parse")

Handles common LLM output quirks:

  • Strips markdown code fences
  • Finds JSON objects embedded in free text
  • Retries with explicit correction on parse failure

Custom exceptions for clear errors:

  • OllamaConnectionError: server not running
  • OllamaModelNotFoundError: model not pulled
  • OllamaJSONError: unparseable output

Network Scan Skill and Subprocess Safety

First implemented skill: nmap service version detection (-sV). Multiple safety layers prevent command injection and ensure reliable operation.

tools/nmap_runner.py

from ipaddress import ip_address

def _validate_target(target):
    ip_address(target)  # raises ValueError
    return target

def run_nmap_service_scan(target, timeout=120):
    safe_target = _validate_target(target)
    if shutil.which("nmap") is None:
        return {"error": "nmap not installed"}

    result = subprocess.run(
        ["nmap", "-sV", safe_target],
        timeout=timeout,
        capture_output=True, text=True)
    return {"services": _parse_nmap_output(...)}

Safety measures (same pattern for future GDB/disassembler wrappers):

  • IP validation: only valid IPv4/IPv6 passes through
  • No shell=True: args as list, no shell injection
  • 120s timeout: prevents hung processes
  • Tool check: verifies nmap exists before execution

Output parsed via regex into structured service dicts:

# 22/tcp open ssh OpenSSH 8.9
r"^\d+/(tcp|udp)\s+(open|filtered|closed)\s+(\S+)\s*(.*)"

Summarizer: Output Compression

After each skill runs, the summarizer uses the local LLM to compress raw output into a concise summary and structured findings. This is what gets stored in memory, not raw tool text.

Example output

{
  "open_ports": [22, 80, 443],
  "services": {
    "22": "ssh",
    "80": "http",
    "443": "https"
  },
  "findings": [
    "3 accessible services detected",
    "HTTP/HTTPS may host web apps"
  ]
}

This is critical for multi-step workflows. When chaining tools (scan, debug, disassemble), raw output would exceed the LLM context window. The summarizer keeps state compact.

Fallback: rule-based extraction when LLM is unavailable. Pipeline never breaks.

System prompt enforces factual output:

"Be concise and factual.
Do NOT fabricate findings not
present in the data."

Memory Manager: Structured State

Stores structured findings across turns. The planner reads a compact snapshot each turn instead of raw tool output, preventing context bloat.

core/memory_manager.py

@dataclass
class MemoryManager:
    current_target: str | None
    discovered_open_ports: list[int]
    discovered_services: dict[str, str]
    findings: list[str]
    attempted_actions: list[ActionRecord]
    condensed_history: list[str]

Key methods:

  • record_scan_results(): merge ports/services, deduplicate
  • record_findings(): add factual observations
  • record_action(): log skill execution and outcome
  • get_context_snapshot(): compact dict for planner

Essential for multi-step chains. When the agent goes from recon to debugging to disassembly, memory carries findings forward without blowing up the context window.

Multi-Turn Workflow

Both turns share the same MemoryManager object. The second turn sees everything the first discovered.

TURN 1: "Scan the target for open services" Planner sees empty memory -> selects network_scan Nmap runs -> summarized -> memory updated Memory after turn 1: ├── current_target: "127.0.0.1" ├── discovered_open_ports: [22, 80] ├── discovered_services: {"22": "ssh", "80": "http"} └── findings: ["2 services detected on target"] same memory object passed to next turn TURN 2: "Summarize the most important findings so far" Planner sees populated memory -> no re-scan needed Produces summary from existing state

For the final, this mechanism supports longer chains: scan, identify service, debug binary, find memory corruption. Each step reads structured results from memory.

Failure Handling and Demo

Graceful degradation at every layer:

  • Ollama down: keyword fallback + rule-based summarization
  • Model not pulled: clear error with fix command
  • Bad JSON: retry with correction prompt
  • nmap missing: detected before subprocess
  • Invalid target: rejected at IP validation
  • Timeout: 120s limit, clean error
  • Unknown skill: validated against registry
Every failure produces a readable message, never a raw traceback.
